# YoCore — v1.8 Design Addendum

**Date:** April 26, 2026
**Status:** ✅ Final — supersedes prior implicit gaps; supplements [YoCore-System-Design.md](../YoCore-System-Design.md)
**Why this exists:** During pre-implementation review, 10 cross-cutting gaps were identified that affect developer workflow, product integrations, or operational safety. Rather than mutate the 4,000-line System Design in place, this addendum captures all 10 in one canonical document.

---

## #1 — Error Code Catalog (mandatory)

All API responses with `4xx`/`5xx` status codes MUST include a stable, programmatic `error` code from a single shared enum. Products consume these via `@yocore/types` ErrorCode.

**Response shape (always):**
```json
{ "error": "EMAIL_NOT_VERIFIED", "message": "Please verify your email before signing in.", "correlationId": "01HXX..." }
```

**Source of truth:** [error-codes.md](./error-codes.md). Enum lives in `packages/types/src/errors/error-codes.ts` and is auto-checked at CI for orphaned codes.

---

## #2 — OpenAPI Spec Generation Strategy

We do **not** hand-write OpenAPI YAML. Every endpoint registers its Zod schemas via `@asteasolutions/zod-to-openapi`; the spec is auto-built from `packages/types`.

- Served at `GET /v1/openapi.json` (cached 5min, public).
- Scalar UI mounted at `/api-docs` (no auth, read-only).
- SDK (`packages/sdk`) is generated/validated against this spec in CI to catch drift.
- Versioning: `info.version` bumps on any breaking change; spec history archived to `docs/openapi-history/v1-YYYYMMDD.json`.

See [openapi-strategy.md](./openapi-strategy.md).

## #6 — `GET /v1/users/me/mfa/status` endpoint

**Gap closed:** End-users had no way to check their MFA enrollment state programmatically; products couldn't surface security UX. Adds:

```
GET /v1/users/me/mfa/status
Authorization: Bearer <jwt>

200 OK
{
  "totp": { "enrolled": true, "verifiedAt": "2026-04-20T10:00:00Z", "lastUsedAt": "2026-04-26T08:30:00Z" },
  "recoveryCodes": { "total": 10, "remaining": 8, "lastRegeneratedAt": "2026-04-20T10:00:00Z" },
  "enforcementRequired": false
}
```

- Reads from `mfaFactors` collection scoped by `(userId, productId)` (productId from API-key middleware).
- Cached 30s in Redis `mfa:status:<userId>:<productId>`.
- Not exposed in audit log (read-only).

## #7 — `GET /v1/entitlements/:workspaceId?includeGrandfatheringInfo=true`

**Gap closed:** Plan-grandfathering was opaque. Products had no way to detect drift between snapshot and current plan limits.

```
GET /v1/entitlements/ws_01HXX?includeGrandfatheringInfo=true

200 OK
{
  "workspaceId": "ws_01HXX",
  "subscriptionId": "sub_01HXX",
  "plan": { "id": "plan_pro", "slug": "pro", "version": 3 },
  "limits": { "maxMembers": 25, "maxProjects": 100, "storageMb": 50000 },
  "isGrandfathered": true,
  "grandfatheringInfo": {
    "snapshotAt": "2025-11-12T10:00:00Z",
    "snapshotPlanVersion": 1,
    "currentPlanVersion": 3,
    "snapshotLimits":  { "maxMembers": 25, "maxProjects": 100, "storageMb": 50000 },
    "currentLimits":   { "maxMembers": 50, "maxProjects": 200, "storageMb": 100000 },
    "driftedFields":   ["maxMembers", "maxProjects", "storageMb"]
  }
}
```

- Source: `subscriptions.planLimitsSnapshot` if present, else live `billingPlans.limits`.
- Cached 60s in Redis `entitlement:<productId>:<workspaceId>`.
- Invalidated on subscription `plan_change`, plan `publish`, plan `archive`.

## #8 — Email Re-enable Mechanism

**Gap closed:** `productUsers.emailDeliverable=false` (set on hard bounce) had no automated rehabilitation. Now:

**Cron `email.deliverability.review`** (daily, 03:00 UTC, idempotent via `cronLocks`):
- Find users where `emailDeliverable=false AND emailDeliverableUpdatedAt < now-90d`.
- Cross-check `emailEvents` for any new hard bounce in last 90d → if any, skip + extend `emailDeliverableUpdatedAt`.
- Else flip `emailDeliverable=true`, log to `auditLogs` action `user.email_deliverable_restored`.

**Manual override (admin):** `POST /v1/admin/users/:id/email-deliverability/reset` (Super Admin only) — flips immediately + audit.

**Schema add:** `productUsers.emailDeliverableUpdatedAt` (timestamp).

## #9 — JWT Blocklist Dual-Check (Redis primary + Mongo fallback)

**Gap closed:** If Redis is down, force-logout via `jwt:blocklist:<jti>` silently fails — token remains valid until natural exp. Mitigation:

**JWT verify middleware order:**
1. Verify signature against keyring (Redis-cached or in-memory).
2. Check Redis `jwt:blocklist:<jti>` — if SET, reject 401.
3. **If Redis is down OR connection error**, fall back to: `sessions.findOne({ jwtId: jti })`. If `revokedAt != null`, reject 401.
4. Else accept.

Fallback adds ~10ms (Mongo round-trip) on Redis failure only. Sentry alert fires when fallback path is taken (indicates Redis incident).

## #10 — Circuit Breaker Prometheus Metrics

**Gap closed:** `opossum` circuit state changes were logged but not exposed as metrics. Adds:

```
yocore_circuit_state{provider="stripe"}      0=closed | 1=open | 2=halfOpen
yocore_circuit_failures_total{provider}       counter
yocore_circuit_successes_total{provider}      counter
yocore_circuit_rejects_total{provider}        counter
```

- Updated via `opossum` event hooks (`open`, `halfOpen`, `close`, `failure`, `success`, `reject`).
- Alert: `yocore_circuit_state == 1 for 60s` → page on-call.
- Per-provider gauges: stripe, sslcommerz, paypal (post-MVP), paddle (post-MVP), resend, ses, s3, kms.
- Exposed at `/metrics` (Prometheus scrape; auth via shared secret header).
