# ADR-009 — Webhook idempotency: Mongo unique constraint as source of truth

**Status:** Accepted

## Context
Stripe (and other gateways) deliver webhooks at-least-once. Same `evt_xxx` may arrive 2-N times. Processing it twice risks: double-activation of a subscription, duplicate invoice marking, etc.

## Decision
Every inbound webhook handler:
1. Insert into `webhookEventsProcessed` with unique index on `{provider, eventId}`. E11000 on duplicate = "already processed" → return 200 immediately.
2. Belt-and-suspenders: Redis `SET lock:webhook:<provider>:<eventId> 1 NX EX 30` BEFORE the Mongo insert (catches burst within sub-second window before Mongo write flushes).
3. Only after both pass, run the actual business logic.

For SSLCommerz (no globally-unique event ID), use merchant-generated `tran_id` (UUID, generated by us at checkout time) as the `eventId`.

## Rationale
- Mongo unique index is durable across Redis failures.
- Redis fast-path keeps p99 latency low (no Mongo write on duplicates).
- Both layers must agree before processing → near-zero double-fire risk.

## Consequences
- `webhookEventsProcessed` has 90d TTL (sufficient for replay window).
- `tran_id` for SSLCommerz must be persisted in `subscriptions.gatewayRefs.sslcommerzTranId` at checkout creation.
- Every gateway integration adds its `provider` value to the dedup table.
