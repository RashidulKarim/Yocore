Part 5 of 6
Architectural Review
Honest assessment of what is well-designed, what are the gaps with severity ratings, and answers to specific architectural questions.
✅ What is well-designed
Provider-agnostic gateway model
paymentGateways.credentialsEncrypted + subscriptions.gatewayRefs — both flexible objects. Adding a 5th gateway is zero schema migration, only a new handler. Highest-leverage decision in the whole design.
Global identity, per-product membership
users is global. productUsers is the junction. Email verification is global — once verified, any new product signup skips it entirely. Solves Flow I cleanly.
Currency-based gateway routing in config
products.billingConfig.gatewayRouting maps { usd: "stripe", bdt: "sslcommerz", default: "stripe" } — business logic as config, not branching code.
Refresh-token families for theft detection
Each session carries a refreshTokenFamilyId. If a revoked token is presented again, the entire family is revoked and user is forced to re-login. Cheap + effective.
All secrets hashed / encrypted at rest
Refresh tokens, invitation tokens, password reset tokens, API secrets — all stored as SHA-256 hashes or AES-256-GCM ciphertext. Raw values only appear in transit.
Per-product billing lifecycle config
Grace period, hold days, warning schedules stored in products.billingConfig — not hardcoded. Each product can have different suspension timelines.
⚠️ Gaps & risks
G1 — Webhook idempotency not enforced at storage layer
🔴 Critical
Stripe/SSLCommerz can deliver the same event twice. Without a unique constraint you can double-activate or double-charge.
Fix: Add a webhookEventsProcessed collection: { provider, eventId, productId, processedAt } with {provider, eventId} unique index. Every handler does insertOne first — if it throws E11000, return 200 immediately.
G2 — Subscription gatewayRefs lacks unique constraints
🔴 Critical
Fix: Sparse unique indexes on gatewayRefs.stripeSubscriptionId, gatewayRefs.paypalSubscriptionId, gatewayRefs.paddleSubscriptionId. Already defined in §1.10.
G3 — No protection against creating two SUPER_ADMINs
🔴 Critical
Fix: Partial unique index: db.users.createIndex({ role: 1 }, { unique: true, partialFilterExpression: { role: "SUPER_ADMIN" } })
G4 — SSO cookie attributes need explicit documentation
🔴 Critical
Fix: Domain=.yo.com; Path=/; HttpOnly; Secure; SameSite=Lax. Cross-apex products use the 60s SSO handoff token in authTokens.type='sso_handoff'.
I1 — billingScope semantics need schema validation
🟡 Important
Fix: Mongo $jsonSchema validator on subscriptions requiring exactly one of subjectUserId / subjectWorkspaceId is non-null, matching the parent product's billingScope.
I2 — Audit log missing SOC 2 fields
🟡 Important
Missing: correlationId (request ID), actor.sessionId, geographic context, before/after snapshots for state changes. Hash chain (prevHash/hash) is proposed but not yet built.
I4 — Permission cache invalidation strategy
🟡 Important
Fix: On workspaceMembers write/update/delete and roles update, publish to Redis pub/sub channel cache:invalidate with key perm:<productId>:<userId>:<workspaceId>. All API nodes subscribe.
N1 — Rate-limit headers on every response
🟢 Nice to have
X-RateLimit-Limit, X-RateLimit-Remaining, X-RateLimit-Reset on all endpoints.
N2 — Idempotency-Key support on mutating calls
🟢 Nice to have
Support Idempotency-Key header on signup, checkout, subscription cancel — safe to retry without side effects.
💳 SSLCommerz: Stripe-as-calendar analysis
Verdict: Yes for MVP, with eyes open.

Pros: Reuses Stripe's mature recurring engine — period math, invoices, dunning hooks — all for free.
Cons: Two systems must stay in sync. If SSLCommerz IPN succeeds but stripe.invoices.pay fails → false payment_failed storm.
Mitigation: Retry stripe.invoices.pay with idempotency key for 24h before alarming.

PayPal: Native recurring — no calendar needed. BILLING.SUBSCRIPTION.RENEWED fires on each cycle.
Paddle: Native recurring + Merchant of Record — handles VAT, refunds, dunning. Best architecture.
